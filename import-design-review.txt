================================================================
インポート設計レビュー：複数クライアント運用・表記ゆれリスク精査
================================================================
対象アプリ：シフト作成サポートアプリ（React + TypeScript + Vite）
調査日：2026-05-12
================================================================


【前提】
- クライアントは複数存在する
- クライアントごとに会期リストのExcel/CSV形式が異なる可能性がある
- こちらのテンプレートを必ずしも使ってもらえるとは限らない
- 会場名は将来的にはマスタ化・プルダウン化したい
- 現実には、独自ファイル由来の表記ゆれを吸収する必要がある
- 既存データや手動編集済みデータは壊してはいけない


================================================================
1. フィールドの責務は破綻していないか
================================================================

基本設計は概ね正しい。ただし3箇所で破綻している。

フィールド             | 設計の意図                         | 現状
-----------------------|------------------------------------|---------------------------
rawSiteName            | 原文保存・復元専用                  | OK。正しく使われている
siteName               | cleanSiteName後の親会場名            | OK。ただしclientName問題と連動
subSiteName            | 物理区画名（独立フィールド）          | 警告：ImportWizard内では
                       |                                    | subSiteNameRaw として保持。
                       |                                    | 正規化後との名前が異なる
displaySiteName        | UI表示専用                          | OK。キーには使われていない
normalizedSiteKey      | グルーピング・類似候補検索用          | OK。ただしapplySiteNormalize
                       |                                    | 呼び出しに完全依存
siteIdentityKey        | importDiff / 再インポート判定の基準  | 警告：同上。かつALIAS_DICT
                       |                                    | 追加時に既存値と食い違う（後述）
clientName             | クライアント名                      | 危険：入力源が3つあり、
                       |                                    | 優先順位が実装場所によって異なる

■ clientName の3つの入力源と競合リスク

  (1) csvImport.ts     : CSVの clientName 列を直接パース
  (2) rawImport.ts:190 : ユーザーがウィザードで入力した fallback 値
  (3) shiftNormalize.ts:209 : applySiteNormalize内でsiteName括弧から自動抽出

  問題：
  rawImport.ts の fallback 値が設定されていると、applySiteNormalize
  内の括弧抽出が無視されないケースがある
  （site.clientName?.trim() || extractedClient の分岐）。

  例：
    ウィザードで誤った fallback 値を入力
    → siteName に括弧が残ったまま clientName だけ上書きされる
    → siteIdentityKey が正しく生成されない


================================================================
2. alias辞書をどの層で適用するのが安全か
================================================================

■ 結論：applySiteNormalize の内部（正規化キー生成時のみ）に閉じ込める。
  displaySiteName・siteName・rawSiteName には絶対に触れない。

現在の実装（normalizeAlias → buildNormalizedSiteKey の内部）は
この方針通りで正しい。

■ 危険なアンチパターン：
  alias を適用した後の文字列を siteName に書き戻すこと。
  rawSiteName が意味を失い、元のクライアントファイルとの突合が
  できなくなる。


================================================================
3. alias辞書を既存データに自動適用する危険
================================================================

■ 最大の地雷。alias辞書にエントリを追加すると、
  既存WorkSiteのsiteIdentityKey と、再インポート時に新規生成される
  キーが食い違い、データが静かに壊れる。

【具体例】
  SITE_NAME_ALIAS_DICT に 'bivi': 'ビビ' を追加した場合：

  既存データの siteIdentityKey = "\0bivi"  （追加前に生成）
  再インポート時の siteIdentityKey = "\0ビビ" （追加後に生成）

  → diffImportBatch が別会場として扱う
  → 全件「追加」判定になる
  → 旧データを削除できず重複登録が発生する

■ 対策なしに alias を追加してはいけない。

  追加するなら：
  (a) localStorage の既存 WorkSite を一括マイグレーションする、または
  (b) 「alias追加後は旧バッチ削除→再インポート」をユーザーに強制する
      フローを設ける


================================================================
4. 同一会場候補提案はどの条件なら表示してよいか
================================================================

■ 現在の実装の問題点（ImportWizard.tsx）

  問題1：閾値 0.5（bigram 50%一致）は緩すぎる
    → 4文字の会場名なら2文字一致で候補表示される

  問題2：findCandidates に渡す入力が rawSiteName（cleanSiteName未適用）
    → "+2名" や "※サテ" が含まれたままスコア計算
    → 正規候補を見逃す可能性がある

  問題3：clientName が候補検索に使われていない
    → 異なるクライアントの同名会場が候補に混入する

■ 推奨条件

  表示してよい条件：
    cleanSiteName(rawSiteName) のbigramスコア ≥ 0.7
    かつ clientName が一致（空同士は一致とみなす）
    かつ subSiteName が一致（空同士は一致とみなす）

  スコア 0.5〜0.7 の場合：
    「低信頼度候補」として別表示・デフォルト非選択


================================================================
5. 自動マージしてはいけないケース
================================================================

自動マージ（ユーザー確認なし）は現状行われていないが、
将来実装する際に禁止すべきケース：

ケース                              | 理由
------------------------------------|-----------------------------------
clientName が異なる                  | 別クライアントの同名会場が統合される
subSiteName が異なる                 | 同会場の別区画（1F・2F等）が統合される
bigramスコアが 0.7 未満              | 誤検知の可能性が高い
既存側が isManuallyEdited: true      | ユーザーが意図して変更した内容を壊す
既存側が source: 'manual'            | 手動作成データを外部ファイルで上書き
日付範囲が全く重ならない             | 別会期・別案件の可能性が高い
時間帯が大きく異なる（2時間以上）    | 同一会場でも別シフト設定の可能性

■ 現状の最大の穴：
  buildImportSites で「新規として登録」を選択した際、
  既存の siteIdentityKey と一致するものがあっても
  現在は無警告で重複登録される。


================================================================
6. importエラー一覧として出すべき項目
================================================================

現在は ParseError（行番号＋メッセージ）のみ出力。
以下を3段階で分けて追加すべき。

■ エラー（スキップ）：
  - 必須列（date / siteName / startTime / endTime）の欠落
  - 日付フォーマット不正
  - 時刻フォーマット不正
  - requiredPeople が 0 以下または非数値

■ 警告（取り込むが明示）：
  - siteName に "+N名" が含まれていた（requiredPeople に加算して除去済み）
  - siteName に "※..." が含まれていた（除去済み）
  - clientName が空で、括弧からも抽出できなかった
  - cleanSiteName の前後で文字列が変化した（変化前→変化後を表示）

■ 確認（ユーザー選択が必要）：
  - siteIdentityKey が既存 WorkSite と一致する
    → スキップ / 上書き / 新規 を選択させる
  - bigramスコアが 0.7以上の候補が1件だけ存在する
    → 既存に紐付けるか確認
  - 候補が複数でスコア差が 0.1 以内
    → 自動選択禁止、ユーザーに選ばせる


================================================================
7. 次に実装すべき最小改修（優先度順）
================================================================

■ 優先度 S（データ破壊リスク・今すぐ対処が必要）

  S-1. buildImportSites での既存重複チェック追加
    内容：siteIdentityKey ベースで既存 WorkSite と照合し、
          「新規登録」選択時でも完全一致があれば警告表示。
          自動マージはしない。
    影響ファイル：ImportWizard.tsx（buildImportSites 関数）

  S-2. findCandidates の入力を cleanSiteName 適用後に変更
    内容：rawSiteName → cleanSiteName(rawSiteName) に変更。
          スコア閾値を 0.5 → 0.7 に引き上げ。
    影響ファイル：ImportWizard.tsx（findCandidates 関数）

■ 優先度 A（表記ゆれ吸収の精度向上）

  A-1. findCandidates に clientName を加える
    内容：simpleNorm(clientName + siteName + subSiteName) で一体評価。
          クライアントが違えば候補に出ない。
    影響ファイル：ImportWizard.tsx（findCandidates 関数）

  A-2. clientName 確定フローの統一
    内容：fallback値と括弧抽出の優先順位を明確化。
          「括弧優先」か「fallback優先」かをコメントまたはUIで明示。
    影響ファイル：shiftNormalize.ts（applySiteNormalize）、
                  rawImport.ts、ImportWizard.tsx

  A-3. applySiteNormalize 呼び出し箇所の一元化
    内容：ImportWizard.tsx と CsvImporter.tsx の2箇所から呼ばれている。
          将来のパス追加時に呼び忘れが起きないよう、
          WorkSite[] を受け取って必ず normalize を適用する
          ゲートウェイ関数を作る。
    影響ファイル：shiftNormalize.ts（新関数追加）

■ 優先度 B（将来のマスタ化への布石）

  B-1. SITE_NAME_ALIAS_DICT のマイグレーション設計
    内容：alias 追加時に既存 WorkSite の siteIdentityKey を
          一括再計算するマイグレーション関数を用意する。
          変更を検知したら「再インポート推奨」バナーを出す。

  B-2. インポートエラー一覧への「警告」カテゴリ追加
    内容：cleanSiteName による変化・clientName 未確定の行を
          警告として黄色で表示する。
          現在のエラー（赤）と視覚的に分離する。


================================================================
まとめ
================================================================

現状の最大リスクは以下の2点：

  (1) 「新規登録」選択時の重複登録が無防備
      → buildImportSites で siteIdentityKey の完全一致チェックがない

  (2) alias辞書追加時の既存キー不整合
      → 追加後に再インポートすると全件「追加」判定になる

どちらも「静かにデータが壊れる」系のバグで、
ユーザーが気づくのが遅れる。

優先度 S の2点を先に塞いでから、
alias辞書・マスタ化の議論に進むのが安全。
